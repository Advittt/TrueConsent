# Generate Mock Consent Form Templates

This guide explains how agents should create and maintain the mock consent form PDFs used for TrueConsent demos.

## Purpose

The files in `consent-form-mocks/` are fictional consent forms for testing the TrueConsent upload, parsing, summarization, and red-flag explanation flow.

They should look realistic enough for a hackathon demo, but they must never be usable as real medical, legal, billing, dental, or surgical documents.

## Design Requirements

Each mock form should:

- Be clearly labeled as fictional or demo-only.
- Be comprehensive enough to exercise summarization and red-flag detection.
- Contain extractable text, not just scanned images.
- Include realistic consent-form sections, such as risks, benefits, alternatives, patient responsibilities, privacy, financial responsibility, and signatures.
- Include blank fields and acknowledgement checkboxes so the PDF looks like a real form.
- Avoid real patient names, facility names, addresses, phone numbers, MRNs, account numbers, or signatures.

## Safety Boundary

Always include a visible demo notice similar to:

> This fictional document is for product demonstration and OCR testing. It should not be used as a real medical, dental, surgical, legal, or billing document.

Do not imply that TrueConsent provides medical, legal, or financial advice. These forms are test fixtures for a consent-form explainer, not authoritative clinical documents.

## How to Add or Edit a Template

The generator uses a single `build(form)` entrypoint that accepts a FORM dict.
Each form is described by a small data structure rather than a bespoke builder
function, with reusable block helpers for the common patterns:

```python
FORM = {
    "filename": "<category>-consent-form.pdf",
    "header":   {"title": ..., "subtitle": ..., "form_num": ..., "rev_date": ...},
    "patient":  [...field_table rows...],
    "clauses":  [(letter, heading, body), ...],   # lettered legal blocks
    "subforms": [{"title": ..., "body": ..., "ack_items": [...]}],  # e.g., anesthesia
    "fine_print": [...],                           # dense 7.5pt paragraphs
    "checklist":  [...],                           # acknowledgement items
}
```

Reusable block helpers live next to the existing primitives:

- `lettered_clauses(items)` — A./B./C. dense legal blocks.
- `subform(title, body, ack_items)` — standalone consent block with its own signature row.
- `fine_print(paragraphs)` — 7.5pt dense paragraph block.
- `inline_initial(line)` — line ending with `______ (initials)`.
- `form_meta_footer(form_num, rev_date)` — `Form CONS-101 | Rev. 03/2026` tag under the title.

To add or edit a template:

1. Edit `tools/generate_mock_consent_forms.py`.
2. Add or update the FORM dict for the target form. For form-specific
   authenticity (e.g., a radiation-dose table for imaging), add a flowable
   directly into a clause body — `lettered_clauses()` accepts mixed strings and
   flowables in the `body` list.
3. Add the form dict to the `for form in (...)` loop at the bottom.
4. Keep the writing realistic, but fictional and generic.
5. Regenerate the PDFs.
6. Verify that the PDFs render and their text can be extracted.
7. Commit both the generator changes and the generated PDFs.

## Verification Checklist

After regenerating the forms, verify:

- The expected PDFs exist in `consent-form-mocks/`.
- Each PDF opens successfully.
- Each PDF has at least 2 pages for a comprehensive demo.
- Text extraction returns meaningful text.
- The demo notice is present in each PDF.
- The layout is readable and not blank, clipped, or malformed.

## Content Guidance

Good mock consent forms should include clauses that give the app useful material to explain:

- What procedure or treatment is being authorized.
- Important risks and possible complications.
- Alternatives, including no treatment.
- Financial responsibility language.
- Privacy and records usage language.
- Patient responsibilities.
- Acknowledgement checkboxes.
- Signature and witness sections.

Use plain but formal language. The forms should be understandable after summarization, but dense enough that the TrueConsent explainer has meaningful work to do.

## Do Not Include

- Real patient data.
- Real clinician or hospital names.
- Emergency contact details.
- Real insurance information.
- Real legal citations unless they are necessary and verified.
- Medical recommendations tailored to a specific person.

## Agent Notes

If the form generator changes, keep the output deterministic where possible. The PDFs are demo fixtures, so stable filenames and stable structure are more useful than randomized content.

If adding a new category, use a filename pattern like:

```text
<category>-consent-form.pdf
```

For example:

```text
dental-implant-consent-form.pdf
imaging-consent-form.pdf
telehealth-consent-form.pdf
```
